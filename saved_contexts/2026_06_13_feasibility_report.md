# Feasibility Report: Re-platforming Excel Scheduler

**Date:** 2026-06-13
**Author:** Claude (analysis based on current `main` @ v0.5.4)
**Question:** Should Excel Scheduler move off the JS/Office.js Excel add-in? Two candidate targets evaluated — (A) a Python LibreOffice Calc plugin, and (B) a standalone SQLite-backed desktop app — both judged against the north-star goal of **embedding an MCP server for AI-assisted scheduling**.

---

## Decision (2026-06-13)

Direction chosen: a standalone **Go** core (Option B backend) delivered primarily *through Claude* (Option D frontend). Concretely:

- **Engine: Go.** All scheduling logic — the four rules and any future ones — lives here. This is the durable, load-bearing IP; everything else is a thin shell over it.
- **Store: SQLite, one file per schedule.** Same "it's just a file" mental model as today's `.xlsx`, but with a real schema (§3.2) instead of blank-cell-delimited columns.
- **MCP server.** Exposes the engine as tools (`list_unassigned_slots`, `get_broken_rules`, `assign_cohort`, `validate`, `explain_conflict`, …) so Claude can read, validate, and mutate schedules.
- **CLI tool (new requirement).** A human-drivable CLI that exercises **the same code paths as the MCP server** — every MCP tool is a thin wrapper over the same Go function the CLI calls. This gives deterministic, human-verifiable testing of every operation **with no Claude and no UI in the loop**, and doubles as the offline/scriptable interface. Invariant: **CLI command ⇄ MCP tool are two front doors to one function; no logic lives in either wrapper.** Verifying a behavior via the CLI verifies it for the MCP path too.
- **One built view: the master grid.** The only UI we commit to hand-building is the editable master schedule grid (the daily driver).
- **Everything else is autogenerated.** Per-cohort calendars, conflict dashboards, and — the common real-world ask — **a pretty class-schedule PDF** are produced on the fly as Claude Live Artifacts from the MCP tools (§4b.6). No React authored for these. If a Claude-free PDF is ever needed, the engine/CLI can render it deterministically from the same data.

The sections below are the analysis that led here; §4b.6 covers the autogeneration mechanism and the discipline that makes it safe.

## 1. What the app actually is today

Before weighing rewrites, it's worth being precise about how small and how spreadsheet-coupled the current app is. That shapes everything below.

**Code surface (the whole thing):**
- `addin/src/taskpane/taskpane.js` (~530 lines) — the entire product logic: `run()` (the rules engine), `clear()`, and the worksheet `onChanged` auto-check plumbing.
- `addin/src/functions/functions.js` (~80 lines) — two custom spreadsheet functions: `FINDCOHORTCLASS` and `FINDCLASSCOHORT` (pure lookups over a passed-in range).
- `addin/src/commands/commands.js` — boilerplate stub (an unused mailbox-notification `action`, leftover from the Office template).
- `taskpane.html` — a single **Auto-check Rules** toggle. That's the entire UI.
- `manifest.xml`, webpack config, Hugo docs site.

**What the app does:**
1. Reads two sheets — `Schedule` (timeslots × classes grid, filled with cohort names) and `Rules` (one rule per column, options stacked below, empty cells used as delimiters).
2. Validates the schedule against four rules: `AllCohorts`, `ClassRequiresTravel`, `CohortBlacklist`, `OneClassAtATime`.
3. Paints offending cells red and attaches a comment listing the broken rules.
4. Offers two lookup functions for building per-cohort "calendar view" sheets.

**The critical architectural fact:** Excel is doing almost all the heavy lifting. The spreadsheet is simultaneously the **data store**, the **grid editor**, and the **reporting surface**. The add-in contributes only ~600 lines of validation + lookup logic on top. The "rules database" is literally columns of cells with a two-blank-cell delimiter convention (see `splitArrayByEmptyStrings`).

This matters two ways:
- The genuinely valuable, portable IP is *tiny* — the rules engine. It would re-implement in well under a day in any language.
- Everything else the user gets "for free" from Excel (grid editing, persistence, formatting, comments, formula recalc) becomes **your problem** the moment you leave a host spreadsheet app.

There is also a strong **platform reality** worth stating up front: the user develops on Linux (CachyOS). **There is no Excel for Linux.** Office.js add-ins only run in Excel on Windows/Mac/web. So today the author cannot even run their own product natively. That alone is a legitimate forcing function for this whole investigation.

## 2. Option A — Python plugin for LibreOffice Calc

### 2.1 Verdict
**Feasible, moderate effort (~1–2 weeks), runs natively on Linux — but inherits the same architectural ceiling and gives the weakest MCP story of the spreadsheet-hosted options.** Good as a *pragmatic lateral move*; weak as a *strategic foundation*.

### 2.2 How the pieces map

| Current Office.js capability | LibreOffice equivalent | Difficulty |
| --- | --- | --- |
| `Excel.run` / read used range values | UNO: `Sheet.getCellRangeByName` / `getDataArray()` | Easy |
| Paint cell red (`cell.format.fill.color`) | `CellProperties.CellBackColor` | Easy |
| Add comment + replies | UNO cell `Annotations` (no threaded replies — flatten to one note) | Easy, minor fidelity loss |
| `onChanged` worksheet event + debounce | `XModifyListener` on the document | Medium (listener lifecycle is fiddly) |
| Task pane with a toggle | **No task pane concept** — use a toolbar button, menu item, or a Sidebar deck | Medium / downgrade |
| Custom functions `FINDCOHORTCLASS`/`FINDCLASSCOHORT` | **The hard part** (see below) | Hard |

### 2.3 The real friction points

1. **Custom spreadsheet functions in Python are genuinely painful.** Calc add-in functions (the `=NAMESPACE.FUNCTION(...)` kind) traditionally require declaring a UNO IDL interface and registering a typelib — the well-trodden path is Java/C++/Basic, not Python. Python *can* expose Calc functions but it's poorly documented and awkward to package. Realistic mitigations:
   - Drop the custom functions and rebuild the per-cohort views as a **macro that writes a view sheet** on demand, or
   - Ship a thin **Basic shim** that delegates to Python, or
   - Replace them with native Calc array formulas (`INDEX`/`MATCH`/`XLOOKUP`).
   This is a real scope item, not a footnote.

2. **No task-pane UI primitive.** The clean little toggle becomes a toolbar button or a Sidebar deck. Sidebar decks are doable but more involved than Office.js's "point the manifest at an HTML page."

3. **Distribution is a `.oxt` extension.** Users install via Tools → Extension Manager. There's no app-store path (the manifest already aspires to a Microsoft Store listing). For a niche school-scheduling tool this is acceptable, arguably simpler, but it's a different distribution story.

4. **Annotations aren't threaded.** The current UX adds a "Rule Infractions:" comment with one reply per broken rule. UNO annotations are flat — concatenate into one note.

### 2.4 MCP story for Option A
Better than Excel, still mediocre. LibreOffice exposes a **UNO remote bridge** — you can start `soffice` with a socket/pipe and connect from an external Python process. So an MCP server *could* drive a live Calc document over UNO. But the model the AI manipulates is still **cells and a blank-delimited rules convention**, not a clean schema. The AI would be reasoning about spreadsheet geography ("column C, two blanks then a group") rather than domain objects. Workable, but you'd likely end up building a domain-model adapter on top of the cells anyway — at which point you've done most of the work of Option B without the benefits.

### 2.5 When Option A is the right call
If the priority is **"keep the spreadsheet-native workflow and just escape the Excel/Linux trap,"** and MCP is a nice-to-have rather than the point, this is the cheapest credible move. The rules engine ports almost verbatim; the main costs are the custom-function workaround and the UI downgrade.

## 3. Option B — Standalone app (SQLite + Go/Rust + modern frontend)

### 3.1 Verdict
**Most work, but the only option that is a true strategic foundation — and clearly the best fit for the MCP goal.** The user's instinct is correct on one point and worth correcting on another:
- ✅ **Correct:** do *not* reimplement a general-purpose spreadsheet or a generic reporting engine. That's a multi-year tar pit.
- ⚠️ **Worth reframing:** this app doesn't *need* a general spreadsheet. It needs a **purpose-built schedule grid** (timeslots × classes, cells hold cohorts) plus a couple of read-only pivot views. That is a constrained, very buildable surface — an off-the-shelf data-grid component gets you ~80% on day one.

### 3.2 Why SQLite-per-schedule is a genuine upgrade
The rules are currently encoded as blank-cell-delimited columns (`splitArrayByEmptyStrings` exists *solely* to parse that). A real schema kills that whole class of fragility:

```
schedule          (id, name, created_at, ...)
timeslot          (id, label, day, period, sort_order)
class             (id, name)
cohort            (id, name)            -- replaces the "AllCohorts" rule list
assignment        (timeslot_id, class_id, cohort_id)   -- a filled grid cell
rule              (id, type, config_json)               -- typed rules, no blank-delimiter hacks
building_group    (...)                                 -- for ClassRequiresTravel
blackout          (cohort_id, timeslot_id)              -- for CohortBlacklist
```

- One file per schedule = trivial save/share/version/backup (it's just a file, same mental model as the `.xlsx`).
- "AllCohorts" stops being a rule and becomes a foreign-key constraint — validation gets simpler, not harder.
- The reporting/pivot views (the per-cohort calendar) become **one SQL query**, not the `FINDCOHORTCLASS` formula gymnastics documented in `setup.md`. This is the part the user feared was "not worth reimplementing" — but in SQL it's *easier* than what exists today, not harder.

### 3.3 Frontend / language choice
The expensive component is the editable grid. Don't hand-roll it.

| Stack | Grid approach | MCP fit | Notes |
| --- | --- | --- | --- |
| **Rust + Tauri** | Web frontend → use a JS data grid (Glide Data Grid, AG Grid, Handsontable) | Excellent (`rmcp`/official Rust MCP SDK maturing) | Tiny binaries, great multi-OS packaging, strong ecosystem alignment |
| **Go + Wails** | Same web-grid approach | Excellent (`mcp-go`, and an official Go SDK exists) | Dead-simple cross-compilation, very fast to build, batteries-included |
| Rust + egui | Native immediate-mode; you build the grid | Good | More grid work; fewer ready components |
| Go + Fyne | Native; weaker grid story | Good | Grid/table widgets are basic |

**Recommendation: Tauri (Rust) or Wails (Go), both with a web-based data grid.** Pick on team comfort. If the deciding factor is *fastest path to a working MCP-enabled app with painless cross-OS builds*, **Go + Wails** is the pragmatic winner; if you want the strongest long-term type-safety/perf story and don't mind a steeper grid setup, **Rust + Tauri**. Either lets you reuse a mature JS grid for editing while keeping the engine + data + MCP server in the compiled backend.

### 3.4 Effort, honestly
- Rules engine port: **~1 day** (it's ~400 lines of straightforward iteration; arguably cleaner against a real schema).
- SQLite schema + persistence + file open/save: ~2–3 days.
- Grid editor wired to the DB (using an off-the-shelf grid): ~1–2 weeks for something genuinely pleasant.
- Pivot/report views: ~2–3 days (SQL + table render).
- MCP server: ~3–5 days (see §4).
- Packaging/signing for 3 OSes, polish: ~1 week, ongoing.

Call it **6–10 weeks** to a solid v1 for one developer — versus 1–2 weeks for Option A. The premium buys you a clean foundation, native Linux/Mac/Windows support, and a first-class MCP integration.

## 4. The deciding lens: MCP for AI-assisted scheduling

This is the stated north star, so it should drive the decision rather than be an afterthought.

**MCP rewards a clean, programmatic domain model and a small set of well-named tools.** It punishes designs where the "state" is tangled up in a host application's UI/document geometry.

Ranking the three worlds by MCP fit:

1. **Standalone app (Option B) — best.** The app owns its data. You embed an MCP server (stdio for a local AI client, or HTTP/SSE) exposing domain tools directly over the SQLite model and the shared rules engine:
   - `list_unassigned_slots`, `get_broken_rules`, `suggest_assignments`, `assign_cohort(timeslot, class, cohort)`, `validate()`, `explain_conflict(cell)`.
   - The loop is clean: **AI proposes → engine validates → DB commits → UI reflects.** The same engine code backs the UI button, the MCP tools, and CI tests. This is exactly the shape MCP is designed for, and it's why the standalone route is strategically correct *even though* it's the most work.

2. **LibreOffice (Option A) — workable but muddier.** An external Python MCP server can reach a live document over the UNO bridge, but it manipulates cells and the blank-delimiter convention. You'd build a domain adapter anyway — duplicating Option B's modeling work without its payoff.

3. **Stay in Excel/Office.js — worst.** An Office.js add-in is a sandboxed webview; it cannot comfortably *host* a local server process. You'd push MCP outside the add-in and drive the workbook via the Graph API or automation against a running Excel instance — heavy, brittle, and Windows/Mac-only. This is the strongest single argument that the *current* platform is a dead end for the actual goal.

**Conclusion of this section:** if MCP-driven AI scheduling is truly the point of the rewrite (not a nice-to-have), it points hard at Option B.

## 4b. Option D — A Claude "Cowork" plugin / MCP App (Claude *is* the frontend)

> **Added after the initial report**, in response to: "what about a Claude Cowork plugin, where the preview pane shows the schedule in tabs and the user works from the chat window?"

### 4b.1 What actually exists (verified mid-2026)
Two shipped Claude surfaces match this idea, and **both are frontends for an MCP server** — i.e. for the exact backend §4 already recommends building:

1. **Claude Cowork** — Anthropic's desktop agentic app (out of preview). Connects to local files/apps; reads/edits/creates files in folders you specify; ties an area of work to a folder + its instructions + task history. UI is a **drag-and-drop grid of terminal / preview pane / diff viewer / chat**, and the **preview pane renders HTML, PDFs, and local app servers**. This is essentially the "preview pane + tabs + chat" layout, already built.
2. **MCP Apps / Interactive Connectors** — an official MCP extension where tools render **interactive React UI inside the chat** (cards, charts, forms, grids, action buttons) instead of returning text. The cleanest realization of "the schedule shown in tabs, driven from the chat window."

### 4b.2 Why this is not a fourth project — it's a different renderer on Option B
Everything I called the "crown jewel" in §4 is unchanged and load-bearing here:
- **Engine** (the four rules) — Phase 0.
- **SQLite domain model** — Phase 1.
- **MCP server** exposing `list_unassigned_slots` / `get_broken_rules` / `assign_cohort` / `validate` / `explain_conflict` — Phase 2.

The "Claude co-work plugin" idea simply **answers the §4 open question "who renders the UI"** differently: instead of building a Tauri/Wails shell, you let Claude be the shell. The user's "at a minimum we set up a connection to whatever app is being used" *is the MCP server boundary* — build Phase 2 and you have that connection, regardless of where the data lives.

Two concrete delivery modes off the **same** MCP server:

| Mode | How the schedule shows up | Best when |
| --- | --- | --- |
| **Cowork** (point it at the project folder) | Cowork runs your local app server → renders in the **preview pane**; or it drives the SQLite directly via your MCP tools + its own file/bash tools. Tabs = preview tabs (master grid / per-cohort calendar / broken-rules). | Power-user, local-files, "operate my project" workflow. |
| **Interactive Connector / MCP App** (hosted or local MCP server) | Your tools render **React UI in-chat** — the schedule grid and the per-cohort/broken-rules views as tabbed components, with assign/validate action buttons. | Drop-in consumer story, no terminal, AI scheduling is the native interaction. |

### 4b.3 Why this is strategically the best fit for the stated goal
The north star was **AI-assisted scheduling**. In Options A/B the AI is a *feature you add to your app*. Here, **your app is a surface inside the AI** — the chat window is the primary interface, the preview/in-chat UI is the schedule, and "propose → validate → commit → reflect" is the native loop with zero glue. It's the tightest possible expression of the goal, and it's **low marginal cost once Phase 2 (the MCP server) exists.**

### 4b.4 Honest tradeoffs
- **You don't own the shell.** Distribution, auth, and availability ride on Claude (Cowork/desktop/web). Users must have Claude. For a niche school tool that's a real adoption gate — though the target user explicitly *wants* AI scheduling, so it's less of a mismatch than it would be for a generic tool.
- **UI is constrained.** MCP App resources are React components rendered as a guest in Claude's layout — excellent for cards/charts/forms and a read-mostly tabbed grid, but a full spreadsheet-grade *editable* grid is more cramped than an owned Tauri window. The Cowork preview-pane-runs-a-local-server route relaxes this (it's just your web app), at the cost of being more "power user."
- **Platform bet / maturity.** These are 2026-era, fast-evolving surfaces. Building your *only* frontend on them is a bet. **Mitigation is structural and already baked in:** keep the engine + SQLite + MCP server as the durable core (they outlive any UI), and treat the MCP App UI as *one* renderer. If the platform shifts, you still have a standalone GUI option off the same backend.

### 4b.5 Verdict
**Pursue this as the Phase 3 frontend — likely the primary one — rather than (or before) a hand-built Tauri/Wails GUI.** It is the lowest-effort, highest-fit way to realize AI-assisted scheduling, *provided* Phases 0–2 are built first. Do not treat it as a shortcut that skips the data model and engine: it depends on them. Build the MCP server once; render it through Cowork and/or an Interactive Connector; keep the owned-GUI route in your back pocket as platform insurance.

### 4b.6 On-the-fly views — author almost nothing (Live Artifacts)
A capability shipped **April 2026** makes the frontend nearly disposable: **Live Artifacts**. These are persistent, interactive HTML/React views rendered in Claude's preview pane that **connect to an MCP server and refresh with live data** when opened — Claude generates them on demand from a plain description, and your server ships **no pre-authored React**. (A plain artifact is a static snapshot; a Live Artifact adds persistent state + an external MCP data connection + the ability to call Claude's API from inside itself.)

So once the engine + rules + tools are solid, Claude can generate the per-cohort calendar, the broken-rules dashboard, "just 3rd grade's Tuesday", "pivot by building", and — **the most common real-world ask, a pretty class-schedule PDF** — each on demand, from the same MCP tools, with no UI code. (Users frequently want nothing more than a clean schedule PDF; that is an autogenerated artifact, not a feature to build.)

**The one discipline that makes this safe:** all logic in the Go engine; tools return clean, complete, *already-computed* authoritative data; the generated view only displays — it never computes. That confines hallucination risk to **layout** (a cell misplaced), not **data** (a wrong cell) — and for a scheduler people act on, that line is the whole ballgame. Return conflicts pre-computed from Go and the view is pure presentation; return raw rows and let Claude derive "who's double-booked" and a regenerated view can quietly be wrong. This is the same invariant as the CLI⇄MCP shared-code-path rule in the Decision: the engine is the single source of truth; every surface (CLI, MCP tool, generated artifact) only renders its output.

**Where a hand-authored component still earns its place** — two limits push a *small* number of views toward a pre-built MCP App component instead of on-the-fly generation:
- **Determinism / variance** — generated views can look or behave a little differently each time. Fine for exploration; less fine for the canonical daily-driver grid.
- **Write-back ergonomics** — *displaying* data on the fly is great; *mutating* it (drop a cohort into a slot) is cleaner as either an explicit chat-driven tool call (`assign_cohort` → artifact refreshes) or a purpose-built component wired to the tools.

Practical split (this is the adopted plan):

| View type | Approach |
| --- | --- |
| Per-cohort calendars, conflict dashboards, ad-hoc slices, **schedule PDFs** | On-the-fly Live Artifacts — **author nothing** |
| The one editable master grid (daily-driver, write-heavy) | The single view we commit to; start chat-driven (edits = tool calls), harden into a component only if it earns it |

Net scope reduction on top of Option B: **ship the Go engine + SQLite + MCP server + CLI, build only the master grid, and let everything else (PDF included) autogenerate.** One caveat — Live Artifacts + a custom MCP server haven't been exercised hands-on here; the write-back ergonomics above are reasoned from how the pieces are documented to work, worth a quick spike once the MCP server exists.

## 5. Recommendation → adopted plan

This is the path the **Decision** (top of report) commits to — Option B's Go core, delivered through Claude (Option D), in phases so value lands early and risk stays low.

The single highest-leverage move is to **build the rules engine as a standalone Go package with its own typed data model** — never welded to a host's cell geometry the way `taskpane.js` is to `Excel.run`. It becomes the single source of truth behind the CLI, the MCP tools, and every generated view.

Phasing:

1. **Phase 0 — Engine (days).** The four rules against a plain Go domain model (classes/cohorts/timeslots/assignments/rules). Unit-test with the doc examples (`ClassRequiresTravel` and `CohortBlacklist` in `rules.md` are ready-made fixtures).
2. **Phase 1 — Store + CLI.** SQLite schema (§3.2), one file per schedule, open/save. A human-drivable **CLI** over the engine (`validate`, `assign`, `list-unassigned`, `report`, …). Testable product core, zero UI, zero Claude.
3. **Phase 2 — MCP server, sharing the CLI's code paths.** Each MCP tool is a thin wrapper over the *same* Go function the corresponding CLI command calls — **CLI command ⇄ MCP tool, one function underneath, no logic in either wrapper.** Human verification via the CLI is therefore verification of the MCP path. Ships before any UI: a Claude client + the CLI is already a usable "AI-assisted scheduler."
4. **Phase 3 — Master grid + autogeneration.** Hand-build only the one view that earns it: the editable master schedule grid. Everything else — per-cohort calendars, conflict dashboards, **the pretty class-schedule PDF users usually want** — is generated on the fly as Claude Live Artifacts off the MCP tools (§4b.6). Author no React for these.
5. **Phase 4 — Packaging / platform insurance.** Distribute the CLI + MCP server; if the Claude-hosted frontend ever needs a fallback, the owned-GUI route (Tauri/Wails over the same engine) stays available.

**Determinism guardrail (carry through every phase):** logic lives only in the Go engine; the CLI, MCP tools, and generated artifacts all *render*, never *compute*. This is what keeps the autogenerated views safe to act on — hallucination risk is confined to layout, not data.

**If budget/time is ever tight and MCP slips to "later":** Option A (LibreOffice Python plugin) remains the honest cheap escape from the Excel/Linux trap — but it's a *lateral move* that inherits the spreadsheet ceiling, and Phase 0's Go engine is **not** reusable there. The decision above forgoes it in favor of the durable core.

### One-line summary
> The valuable IP is the rules engine. Build it in Go over SQLite, expose it once through a CLI and an MCP server that share the same code paths (so a human can verify every path without Claude or a UI), hand-build only the master grid, and let Claude autogenerate everything else — including the schedule PDF people actually ask for.

## Appendix: risks & open questions
- **Master-grid UX parity (now the only built view).** Users expect copy/paste, fill-down, keyboard nav. Scope is much smaller than a full app — just the one grid — but it's still the main UX risk; budget polish time there.
- **CLI ⇄ MCP shared-path testing.** The payoff of one-function-two-front-doors is only real if it's enforced. Decide the seam early (tools/CLI both call `engine.AssignCohort(...)` etc.), and make the CLI the canonical test surface so MCP coverage comes for free.
- **PDF path: generated vs deterministic.** The common "pretty schedule PDF" can come from Claude (Live Artifact → export) *or* a deterministic engine/CLI exporter. Start with the generated path (zero code); add a Go/CLI PDF exporter only if users need it Claude-free or pixel-stable.
- **Live Artifacts + custom MCP server — unverified here.** Confirm write-back ergonomics and refresh behavior with a small spike once Phase 2 exists, before committing the master grid's interaction model.
- **Migration path for existing `.xlsx` workbooks.** A one-time importer (read the `Schedule`/`Rules` sheets, reusing the blank-delimiter parsing already implemented) would smooth adoption. Small, worth doing — likely a CLI subcommand.
- **MCP transport choice.** stdio (local desktop AI clients) vs HTTP/SSE (remote/multi-client). Likely stdio first; design tools transport-agnostically.
- **Who consumes MCP?** Cowork (local files) vs an Interactive Connector vs claude.ai. Affects whether the server is spawned locally or hosted — keep the Go binary runnable both ways.
